## 1. New data → accuracy (more signal into the models)
- ☐ **P3** **Vegas game/team totals as a model feature — BLOCKED on data, not buildable now.** Assessed: **0 games store any historical implied-team-total / game total** (`games.context` holds only model outputs). MLB fetches no game-total source (only player-prop odds → `market_odds`); NBA fetches ESPN totals *live* for the winner model but never persists them, and the NBA season just ended. No historical column to train on. To enable: (1) persist `market_total`/`implied_team_total` per game going forward (MLB needs a new game-odds fetch), (2) accrue a season, (3) assess vs prop residuals + build. Months out; low priority (MLB models are data-saturated).
- ☐ **P3** **Pitcher velocity & pitch-mix trends** — **data-gated, not pursued.** No Statcast pitch-level ingest exists (verified: 0 velo/pitch-type keys in stats/derived) — like team-defense's OAA, the true signal needs a whole new data source for an expected-marginal gain on already-saturated models. Don't build without a cheaper path.
- ☐ **P3** **Times-through-order penalty** — **assessed, neutral.** Box-score TTO proxies (3rd-time-through PAs, times-through-order = batters_faced/9) add nothing on the new `hits_allowed`/`earned_runs` models (−0.2% / −0.06%) — they're redundant with the `batters_faced` features those models already use (a tree already learns the depth threshold). A true per-PA TTO signal needs a play-by-play ingest, for likely-negligible pitcher-specific gain (the universal TTO effect is already in the run environment). Don't pursue.
- ☐ **P3** **Bullpen rest / availability** — **no applicable target, not pursued.** Bullpen fatigue moves team run totals + reliever lines — markets we don't model (we price batter + starter player props). The only tangential link (opposing-pen fatigue → a batter's hits) is a diluted late-game effect on full-game totals; given team-defense (+0.06%) and TTO (−0.2%) both came back neutral on these same models, it's not worth the derivation.
- ☐ **P3** **Team defense (OAA / DRS)** — **assessed, neutral.** A derivable team Defensive-Efficiency (DER) proxy was tested against the new `hits_allowed` model: the model *uses* it (top feature by gain) but it moves test MAE only **+0.06%** — redundant, because a pitcher's own rolling hits-allowed already encodes the defense behind them. True Statcast OAA isn't ingested (no fielding source) and would need a new ingest for near-zero expected upside. Don't pursue without a cheaper signal.
- ☐ **P3** **NBA referee tendencies** — **data-gated + noise prior, not pursued.** No referee/crew/officials ingest exists anywhere in the pipeline (verified) — testing it needs a brand-new data source. And the precedent is decisive: the MLB **umpire** assessment found the home-plate ump's K-residual beyond the model has split-half reliability **r=+0.095** — officials' marginal effect doesn't replicate (noise). NBA crew foul-rate residual is the same shape; the model already has the dominant pace/usage drivers. Don't build the ingest for near-zero expected signal.
- ☐ **P3** **Travel / rest / time-zone fatigue** — **redundant, not pursued.** `days_rest` is already a feature in every model, and the NBA back-to-back signal already came back neutral. A schedule-density/travel proxy is a function of the same rest information the models already use; with team-defense + TTO both confirming the saturation pattern, no separate gain to expect. (`teams` has only `city`, no coords, so true travel-distance would also need geocoding.)
- ☐ **P3** **Late scratches / beat-writer news** — **assessed: immaterial + already mitigated; predictive version data-gated.** Measured the actual cost across all settled picks: MLB no-shows (47) are all auto-**voided** at settlement (no box-score row → zero P&L), and genuine harmful cases total **3 lost legs ever** (all fringe role players on one 2026-05-30 NBA slate, plus 1 incidental under-win). Suppression already filters known "Out" players, so a pick only reaches a scratch the feed *missed*. Shipped a `scratch_picks` regression-guard in `data_audit` (lost legs in the last 14d on a player who logged <5 min / did_play=false) so the digest catches it if suppression ever breaks. The predictive cross-check-vs-confirmed-lineups version is **data-gated**: no confirmed-lineup feed exists, and `player_injuries` retains only a shallow ~1-month snapshot window — not enough to train on. Not worth a paid lineup feed for ~0 ROI.
- ☐ **P3** **Multi-book consensus** — **already implemented; widening is API-gated.** `predict_today._market_over_prob_for_player` already averages the no-vig `market_over_prob` across *all* books present per player/game, and the fetcher (`historical_odds.SHARP_BOOKS`) already requests 5 sharp books (DK, FD, BetMGM, William Hill, …). But only **DraftKings + FanDuel** ever return player-prop data on the current Odds API plan (verified: market_odds holds exactly those 2 books across 54k rows), so the consensus is already as wide as the data allows. Adding books is gated on API coverage, not code — nothing to build until more books expose player props.

## 2. Model / analytics
- ☐ **P3** **Model ensembling / stacking** — **assessed, neutral; lane closed.** Built `props.models.ensemble_eval` (distribution-preserving: every member outputs a Poisson λ so the CDF+isotonic step survives) and tested two ensembles per stat against the +0.5% A/B gate. Nothing cleared it: **seedbag** (avg of 5 seed/feature-fraction-varied GBMs) gained +0.01% hits / −0.05% total_bases / +0.11% points / +0.22% assists; **glmblend** (GBM ⊕ linear Poisson GLM, weight tuned on val) either collapsed to pure-GBM (w=1.0, +0.00%) or only nudged up by going pure-linear (hits w=0 → +0.40%, which breaks the λ interpretation and still misses the bar). The single calibrated Poisson GBM is at the MAE floor — stacking adds <0.25%, not worth the multi-model inference + recalibration burden. (The ensemble that *does* pay off — model⊕market blending — is already shipped in `blend_weights.py`.) Re-runnable via the harness as data grows / new markets land.
- ◧ **P2** **Probability calibration + Brier decomposition — shipped; edge search refocused on the market side.** A Murphy Brier decomposition on 1,867 settled PP picks separated calibration error from predictive resolution and showed the model's usable signal lives in *ordering* picks, not in the probability *magnitude* (raw confidence ran ~12pts hot — e.g. an 0.87 band realized ~0.47). **Fix shipped (commit 17d4144):** every Sleeper `+EV iff prob×payout>1` decision now calibrates first (`calibrate(model_prob)`, live Platt a≈0.29) across `repo._is_recommended` / `repo._sleeper_roi` / `odds_track.load_sleeper_picks` — rejecting ~48 of 51 false-+EV picks (they only cleared on inflated confidence); PP rec-tier untouched (its cutoff is monotonic → calibration-invariant). `odds_track.MIN_TIER_N=30` gates the ROI verdict ("building" below it, since a 3-pick run collapses the normal-approx CI to a false-certain ±100%). Method chosen by a temporal-OOS bake-off on held-out PP picks (leak-free). With the model's confidence now honestly calibrated, the **active edge frontier is market inefficiency** — the arbitrage layer in the next item, in live forward-testing.
- ◧ **P1** **Market-arbitrage finder (Sleeper vs sharp) — LIVE; forward-testing (current focus).** `props.picks.sleeper_arb` bets Sleeper hits/TB where `sharp_true_prob × sleeper_payout > 1` — model-*independent*: it exploits **market** mispricing, not the model. Sharp prob = the LIVE (pre-game) DK/FD no-vig consensus, converted to Sleeper's line via the market-implied Poisson mean; the finder self-guards to games ≥30min out so it's leak-free even if run mid-slate. Persists to `sleeper_arb` (migration 0015), posts a Discord digest with the running ROI, gated by `MIN_TIER_N=30`. **A rigorous clean backtest** (odds re-snapshotted at each game's start−2h) put the +EV tier at **+6.9% vs a −11.7% bet-everything baseline** — an early positive signal, now under live forward-testing to see whether it holds past n≈150. (The naive first pass showed a spurious +30% — caught as a **data leak**: `historical_odds` snapshots at 22:00 UTC = post-game for day games, AUC 0.77 → the outcome was baked in; the finder never uses that path.) **Scope:** MLB hits/TB (DK/FD don't price HR on this plan; NBA back in Oct).
- ☐ **P3** **Playoff vs regular-season model split** — **assessed: real shift, but the model already absorbs it; marginal + seasonal.** The distribution shift is real — the same NBA players average **~16% fewer points / 20% fewer assists / 13% fewer rebounds** in playoffs (n=182 paired players, ≥20 reg & ≥5 PO games). BUT the live model's measured bias on playoff games is tiny (points +0.06, assists +0.08, rebounds +0.32 over 182/63/127 stored playoff predictions) because it predicts off *rolling* features that track the suppression as a series progresses. Only rebounds shows mild over-prediction. With a small playoff sample (~92 games, 2024-25) and NBA off-season (no near-term relevance), a static `is_playoff` feature would mainly help series-openers + the rebounds bias. REVISIT in October (NBA return) if wanted; not worth building now.

## 4. Ops / automation & data integrity
- ☐ **P3** **PrizePicks placeholders — rechecked: mostly expected, small genuine miss, benign.** The backfill did NOT change the rate — it was never a coverage problem (real games exist for those dates). The "~274k lines/14d" alarm was snapshot-inflated (~10×; actual ~2.4k distinct MLB props/day). Breakdown: **WNBA/NHL placeholders ≈95% untracked players** — international/exhibition events PrizePicks lists that aren't real WNBA/NHL games (correctly unresolvable). **MLB**: of placeholder players on a sample day, ~65% had no real game that day (off-day/cross-date/combo — correct), ~35% did play (genuine `resolve_game_id` miss, most likely **doubleheaders**, which it deliberately keeps as placeholder when >1 exact-date match). Only **1 pick ever** landed on a placeholder → zero recommendation impact. If pursued: add time-of-day disambiguation for doubleheaders + filter PrizePicks' non-league/exhibition events. Low priority.
